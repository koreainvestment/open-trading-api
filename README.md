# Auto Trading System

## 1. `presen` branch의 구조

`presen` 브랜치는 발표 및 제출용으로 정리한 브랜치입니다.

기존 한국투자증권 Open API 샘플 저장소에서 자동매매 시스템 실행에 필요한 부분만 남기고, 불필요한 샘플 코드와 보조 폴더는 제거했습니다.

```text
open-trading-api/
├── README.md
├── images/
│   ├── execution_log.png
│   └── account_status.png
├── .gitignore
└── samsung_auto_trader/
    ├── main.py
    ├── config.py
    ├── auth.py
    ├── api_client.py
    ├── market_data.py
    ├── account.py
    ├── orders.py
    ├── trader.py
    ├── logger.py
    └── requirements.txt
```

각 항목의 역할은 다음과 같습니다.

| 항목 | 설명 |
|---|---|
| `README.md` | 제출을 위해 작성한 시스템의 구조 |
| `images/` | 결과 이미지 저장 폴더 |
| `.gitignore` | 토큰, 캐시, 환경변수, Python 임시파일 등이 GitHub에 올라가지 않도록 관리하는 파일 |
| `samsung_auto_trader/` | 실제 자동매매 시스템 코드가 들어 있는 폴더 |

`token_cache.json`은 실행 중 생성될 수 있는 접근 토큰 캐시 파일입니다.  
보안상 업로드하지 않고 `.gitignore`를 통해 숨겼습니다.

## 2. `samsung_auto_trader` 폴더 구조

```text
samsung_auto_trader/
├── main.py              # 프로그램 실행 시작점
├── config.py            # API 주소, 계좌번호, 환경변수 설정
├── auth.py              # 접근 토큰 발급 및 캐시 관리
├── api_client.py        # API 요청 공통 처리 및 오류 메시지 처리
├── market_data.py       # 삼성전자 현재가 조회
├── account.py           # 계좌 현금 및 보유 주식 조회
├── orders.py            # 매수/매도 주문 요청
├── trader.py            # 전체 자동매매 흐름 제어
├── logger.py            # 로그 출력 설정
└── requirements.txt     # 필요한 Python 패키지 목록
```

실제 자동매매 흐름은 `trader.py`가 중심이 되어 다른 파일들의 기능을 순서대로 호출하는 방식으로 작동합니다.

## 3. `samsung_auto_trader` 코드 설명 및 역할

### 3.1 `main.py`

`main.py`는 프로그램의 실행 시작점입니다.

이 파일은 실제 매수·매도 판단을 직접 수행하지 않습니다.  
대신 설정을 불러오고, 자동매매 객체를 만든 뒤, 자동매매 루프를 실행하는 역할을 합니다.

실행 흐름은 다음과 같습니다.

```text
1. logger 설정
2. 설정값 로드
3. StockTrader 객체 생성
4. trader.run() 실행
```

구조적으로 보면 다음과 같습니다.

```text
main.py
→ load_config()
→ StockTrader(config)
→ trader.run()
```

이렇게 분리한 이유는 프로그램의 시작 부분과 실제 매매 로직을 구분하기 위해서입니다.  
실제 자동매매 판단은 `trader.py`에서 담당하고, `main.py`는 실행을 시작하는 최소한의 역할만 수행합니다.

---

### 3.2 `config.py`

`config.py`는 자동매매 시스템 실행에 필요한 설정값을 관리합니다.

이 파일에서 관리하는 주요 설정은 다음과 같습니다.

```text
계좌번호
App Key
App Secret
토큰 발급 URL
모의투자 API 서버 URL
반복 실행 간격
거래 시작 시간
거래 종료 시간
API endpoint 경로
```

API Key와 App Secret은 코드에 직접 작성하지 않고 환경변수에서 불러오도록 구성했습니다.

필요한 환경변수는 다음과 같습니다.

```text
GH_ACCOUNT      # 계좌번호
GH_APPKEY       # 한국투자증권 Open API App Key
GH_APPSECRET    # 한국투자증권 Open API App Secret
```

`config.py`의 핵심 역할은 다음과 같습니다.

```text
환경변수 읽기
→ 필수 값 존재 여부 확인
→ AppConfig 객체 생성
→ 다른 파일에서 설정값을 사용할 수 있도록 전달
```

또한 이 파일에서는 API endpoint도 관리합니다.

```text
price_inquiry      # 현재가 조회
balance_inquiry    # 계좌 잔고 조회
holdings_inquiry   # 보유 가능 수량 조회
order_submission   # 주문 요청
```

### 3.3 `auth.py`

`auth.py`는 한국투자증권 Open API의 접근 토큰을 관리하는 파일입니다.

Open API를 사용하려면 먼저 접근 토큰이 필요합니다.  
이 프로젝트에서는 매번 새 토큰을 발급받지 않고, 기존 토큰이 유효하면 재사용하도록 만들었습니다.

전체 흐름은 다음과 같습니다.

```text
1. token_cache.json 존재 여부 확인
2. 기존 토큰이 유효한지 확인
3. 유효하면 기존 토큰 재사용
4. 없거나 만료되었으면 새 토큰 요청
5. 새 토큰을 token_cache.json에 저장
6. access token 반환
```

#### 3.3.1 토큰 캐시 불러오기

기존에 저장된 `token_cache.json`이 있으면 파일을 읽고, 토큰 발급 시각과 만료 시간을 확인합니다.

```text
토큰 파일 존재
→ issued_at 확인
→ expires_in 확인
→ 현재 시각과 비교
→ 아직 유효하면 재사용
```

#### 3.3.2 새 토큰 발급

캐시된 토큰이 없거나 만료되었으면 한국투자증권 Open API에 새 토큰을 요청합니다.

요청에는 다음 정보가 필요합니다.

```text
grant_type
appkey
appsecret
```

토큰 요청이 성공하면 `access_token`을 받아옵니다.

#### 3.3.3 토큰 저장

새로 발급받은 토큰은 `token_cache.json`에 저장합니다.

저장되는 정보는 다음과 같습니다.

```text
access_token
issued_at
expires_in
```

단, `token_cache.json`은 보안상 GitHub에 업로드하지 않습니다.  
실행 중 생성되는 임시 인증 파일로 취급합니다.

---

### 3.4 `api_client.py`

`api_client.py`는 한국투자증권 Open API 요청을 공통적으로 처리하는 파일입니다.

이 파일의 핵심 역할은 다음과 같습니다.

```text
API URL 구성
인증 헤더 구성
GET 요청 처리
POST 요청 처리
HTTP 오류 처리
API 업무 실패 응답 처리
초당 거래건수 초과 오류 처리
JSON 응답 파싱
```

그리고 API 요청 자체의 공통 처리 방식은 `api_client.py`에서 관리합니다.

특히 자동매매 프로그램에서는 API 호출이 반복되기 때문에 오류 처리가 중요합니다.  
예를 들어 초당 거래건수를 초과하면 API가 요청을 거절할 수 있습니다.  
이 경우 일정 시간 대기한 뒤 재시도하도록 처리해야 합니다.

따라서 `api_client.py`는 단순한 요청 함수가 아니라, 자동매매 시스템의 API 통신 안정성을 담당하는 파일입니다.

---

### 3.5 `market_data.py`

`market_data.py`는 삼성전자 현재가를 조회하는 파일입니다.

자동매매에서 가장 먼저 필요한 정보는 현재 가격입니다.  
현재가가 있어야 매수 가격과 매도 가격을 계산할 수 있기 때문입니다.

이 파일의 흐름은 다음과 같습니다.

```text
1. 삼성전자 종목코드 전달
2. 현재가 조회 API 호출
3. API 응답 수신
4. 응답 데이터에서 현재가 추출
5. 현재가를 정수형으로 변환
6. trader.py로 반환
```

현재가 조회 대상은 삼성전자이며, 종목코드는 다음과 같습니다.

```text
005930
```

`market_data.py`는 직접 매수·매도를 판단하지 않습니다.  
이 파일은 오직 시장 가격 데이터를 가져오는 역할만 담당합니다.

```text
market_data.py = 현재가 데이터 제공자
```

---

### 3.6 `account.py`

`account.py`는 계좌 정보를 조회하는 파일입니다.

자동매매 시스템은 단순히 현재가만 안다고 주문할 수 없습니다.  
매수를 하려면 현금이 충분해야 하고, 매도를 하려면 주식을 보유하고 있어야 합니다.

따라서 `account.py`는 다음 정보를 확인합니다.

```text
주문 가능 현금
삼성전자 보유 수량
계좌 상태
```

이 파일의 흐름은 다음과 같습니다.

```text
1. 계좌번호와 종목코드 전달
2. 계좌 조회 API 호출
3. 주문 가능 현금 조회
4. 삼성전자 보유 수량 조회
5. AccountInfo 형태로 결과 반환
```

자동매매 판단에서 중요한 조건은 다음 두 가지입니다.

```text
매수 가능 여부 = 주문 가능 현금이 충분한가?
매도 가능 여부 = 삼성전자를 보유하고 있는가?
```

이 정보는 `trader.py`에서 매수·매도 조건을 판단하는 데 사용됩니다.

```text
account.py = 계좌 상태 확인 담당
```

---

### 3.7 `orders.py`

`orders.py`는 매수·매도 주문을 요청하는 파일입니다.

자동매매 시스템에서 주문 요청은 실제 API와 연결되는 중요한 부분입니다.  
이 파일은 주문에 필요한 정보를 한국투자증권 Open API 형식에 맞게 구성한 뒤, 주문 요청을 보냅니다.

주문 요청에 필요한 주요 정보는 다음과 같습니다.

```text
계좌번호
종목코드
주문 수량
주문 가격
매수/매도 구분
```

이 프로젝트에서는 지정가 주문 방식을 사용합니다.

```text
매수 주문 → 지정가 매수
매도 주문 → 지정가 매도
```

매수와 매도는 서로 다른 거래 ID를 사용합니다.

```text
매수 주문 → VTTC0802U
매도 주문 → VTTC0801U
```

주문 요청 후에는 API 응답을 해석해 다음 정보를 확인합니다.

```text
주문 성공 여부
주문번호
응답 메시지
```

이 파일은 주문 판단을 직접 하지 않습니다.  
매수할지, 매도할지는 `trader.py`가 판단합니다.  
`orders.py`는 판단 결과를 바탕으로 실제 주문 요청만 담당합니다.

```text
orders.py = 주문 요청 담당
```

---

### 3.8 `trader.py`

`trader.py`는 자동매매 시스템의 핵심 제어 파일입니다.

이 파일은 다른 파일들의 기능을 순서대로 호출하면서 전체 자동매매 흐름을 구성합니다.

즉, `trader.py`는 직접 모든 기능을 수행하는 파일이라기보다, 각 기능을 연결하는 조정자 역할을 합니다.

전체 흐름은 다음과 같습니다.

```text
1. 접근 토큰 확인
2. 거래 가능 시간 확인
3. 현재가 조회
4. 계좌 상태 조회
5. 매수 가격과 매도 가격 계산
6. 매수 가능 여부 판단
7. 매도 가능 여부 판단
8. 조건이 맞으면 주문 요청
9. 주문 후 계좌 상태 재조회
10. 체결 여부 간단 확인
11. 일정 시간 대기 후 다음 사이클 반복
```

`trader.py`가 호출하는 파일들은 다음과 같습니다.

```text
auth.py         → 접근 토큰 확인
market_data.py  → 현재가 조회
account.py      → 계좌 상태 조회
orders.py       → 주문 요청
logger.py       → 실행 로그 출력
```

#### 3.8.1 거래 시간 확인

자동매매가 아무 시간에나 작동하면 주문 가능 시간이 아닐 때 계속 API 요청을 보낼 수 있습니다.  
이를 막기 위해 거래 가능 시간을 확인합니다.

이 프로젝트의 거래 시간은 다음과 같이 설정되어 있습니다.

```text
09:10 ~ 15:30
```

또한 실행 환경이 UTC 시간을 사용할 수 있기 때문에, 한국 시간 기준으로 거래 시간을 판단하도록 설정했습니다.

```text
Asia/Seoul 기준 시간 사용
```

#### 3.8.2 현재가 조회

거래 시간이 맞으면 삼성전자 현재가를 조회합니다.

```text
market_data.py 호출
→ 현재가 반환
```

이 현재가를 기준으로 매수 가격과 매도 가격을 계산합니다.

#### 3.8.3 계좌 상태 조회

현재가 조회 후 계좌 상태를 확인합니다.

```text
account.py 호출
→ 주문 가능 현금 확인
→ 보유 주식 수량 확인
```

이 정보는 매수와 매도 가능 여부를 판단하는 데 사용됩니다.

#### 3.8.4 매수·매도 조건 판단

`trader.py`는 현재가와 계좌 정보를 바탕으로 매수·매도 가능 여부를 계산합니다.

```text
매수 가능 여부 = 주문 가능 현금이 매수 주문 금액보다 큰가?
매도 가능 여부 = 보유 수량이 주문 수량보다 크거나 같은가?
```

#### 3.8.5 주문 요청

조건을 만족하면 `orders.py`를 통해 주문을 요청합니다.

```text
매수 가능 → 지정가 매수 주문 요청
매도 가능 → 지정가 매도 주문 요청
```

한 사이클에서 매수와 매도는 각각 최대 1번만 수행하도록 제한했습니다.

#### 3.8.6 주문 후 계좌 재조회

주문 요청 후에는 계좌 상태를 다시 조회합니다.

이를 통해 주문 전과 주문 후의 현금 및 보유 수량 변화를 비교합니다.

```text
주문 전 계좌 상태
주문 후 계좌 상태
변화 여부 확인
```

이 방식으로 체결 여부를 간단히 추정합니다.

### 3.9 `logger.py`

`logger.py`는 프로그램 실행 로그를 출력하는 파일입니다.

자동매매 프로그램에서는 실행 중 어떤 판단이 이루어졌는지 확인하는 것이 중요합니다.  
따라서 로그를 통해 현재 상태와 API 응답 결과를 확인할 수 있도록 했습니다.

로그를 통해 확인할 수 있는 내용은 다음과 같습니다.

```text
프로그램 시작 여부
토큰 재사용 여부
현재가 조회 결과
계좌 조회 결과
매수/매도 판단 결과
주문 요청 결과

logger.py = 실행 흐름 확인 및 디버깅 담당

`requirements.txt`는 이 프로젝트 실행에 필요한 Python 패키지를 정리한 파일입니다.

requests

`requests`는 한국투자증권 Open API에 HTTP 요청을 보내기 위해 사용합니다.

패키지는 다음 명령어로 설치할 수 있습니다.

pip install -r requirements.txt
```

## 4. 실행 방법

프로젝트 루트에서 `samsung_auto_trader` 폴더로 이동합니다.

```bash
cd samsung_auto_trader
```

필요한 패키지를 설치합니다.

```bash
pip install -r requirements.txt
```

프로그램 실행에 필요한 환경변수는 다음과 같습니다.

```text
GH_ACCOUNT
GH_APPKEY
GH_APPSECRET
```

로컬 환경에서 직접 실행하는 경우에는 다음과 같이 환경변수를 설정합니다.

```bash
export GH_ACCOUNT="계좌번호"
export GH_APPKEY="앱키"
export GH_APPSECRET="앱시크릿"
```

GitHub Codespaces를 사용하는 경우에는 위 값을 Codespaces Secrets에 등록하여 사용할 수 있습니다.

프로그램을 실행합니다.

```bash
python main.py
```

실행 중인 프로그램을 멈추려면 터미널에서 `Ctrl + C`를 누릅니다.

실행 후 자동매매 프로그램은 다음 순서로 작동합니다.

```text
토큰 확인
→ 거래 시간 확인
→ 현재가 조회
→ 계좌 조회
→ 매수/매도 판단
→ 주문 요청
→ 주문 후 계좌 재조회
→ 다음 사이클 대기
```

## 5. 매수·매도 로직

이 프로젝트는 복잡한 투자 전략이 아니라, 자동매매 시스템의 기본 구조를 보여주기 위한 단순한 매매 로직을 사용합니다.

기본 설정은 다음과 같습니다.

```text
대상 종목 = 삼성전자
종목 코드 = 005930
주문 수량 = 1주
매수 가격 = 현재가 - 1,000원
매도 가격 = 현재가 + 1,000원
```

### 5.1 매수 로직

매수 가격은 현재가보다 1,000원 낮게 설정합니다.

```text
매수 가격 = 현재가 - 1,000원
```

매수 조건은 다음과 같습니다.

```text
주문 가능 현금 ≥ 매수 가격 × 주문 수량
```

이 조건을 만족하면 지정가 매수 주문을 요청합니다.

```text
현금 충분
→ 지정가 매수 주문 요청
```

현금이 부족하면 매수 주문을 건너뜁니다.

```text
현금 부족
→ 매수 주문 생략
```

### 5.2 매도 로직

매도 가격은 현재가보다 1,000원 높게 설정합니다.

```text
매도 가격 = 현재가 + 1,000원
```

매도 조건은 다음과 같습니다.

```text
보유 주식 수량 ≥ 주문 수량
```

이 조건을 만족하면 지정가 매도 주문을 요청합니다.

```text
보유 수량 있음
→ 지정가 매도 주문 요청
```

보유 수량이 부족하면 매도 주문을 건너뜁니다.

```text
보유 수량 부족
→ 매도 주문 생략
```

### 5.3 한 사이클당 주문 제한

이 프로젝트에서는 한 사이클 안에서 매수와 매도를 각각 최대 1번만 요청하도록 설정했습니다.

```text
한 사이클 최대 매수 주문 = 1회
한 사이클 최대 매도 주문 = 1회
```

이 제한을 둔 이유는 반복 실행 중 과도한 주문 요청이 발생하는 것을 막기 위해서입니다.

### 5.4 전체 매매 흐름

한 번의 사이클은 다음 순서로 진행됩니다.

```text
1. 거래 가능 시간인지 확인
2. 삼성전자 현재가 조회
3. 계좌 현금 및 보유 수량 조회
4. 매수 가격 계산
5. 매도 가격 계산
6. 매수 가능 여부 판단
7. 매도 가능 여부 판단
8. 조건 충족 시 지정가 주문 요청
9. 주문 후 계좌 상태 재조회
10. 다음 사이클까지 대기
```
